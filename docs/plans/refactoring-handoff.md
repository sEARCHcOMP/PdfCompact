## 0. まず最初に読め: このプロジェクトの「物理法則」

知らずに触ると確実に事故る制約。これは好みでも方針でもなく、**動かすための前提**や。

### 法則1: 成果物は単一HTMLでなければならない
- 利用者(建設会社の非エンジニア)は `PDF Compact.exe`/`.bat` をダブルクリックするだけ。サーバーもインストールも無い。
- auto-update は GitHub の `version.json` を見て `pdf_compact_bundle.html` 1ファイルを差し替える仕組み。**ファイルが1つやから成立しとる。**
- → リファクタで src/ に分割しても、**最終成果物は必ず1ファイルに連結し直す**。これを忘れた瞬間に製品が壊れる。
- bundler/webpack/vite を入れたくなるが**入れるな**。理由は法則2。

### 法則2: ツールチェーンの寿命 < このツールの寿命
- これは社内の底上げツールで、5年10年と現場で使われる想定。一方フロントのビルドツールは2-3年で陳腐化する。
- 依存を増やすほど「数年後に誰も再ビルドできない」リスクが上がる。
- → ビルドは **node の標準APIだけで書いた「連結スクリプト」** に留める。minify すらしない(出力HTMLが読める=デバッグできる状態を保つ)。
- pdf.js は CDN、pdf-lib/fontkit/jsPDF/JSZip は同梱。この構成を増やさない。

### 法則3: ワーキングコピーは CRLF
- git が autocrlf でチェックアウト時に CRLF 化しとる。
- node でファイルを編集するときは必ず `.replace(/\r\n/g,'\n')` で LF 正規化 → 編集 → `.replace(/\n/g,'\r\n')` で CRLF 書き戻し。
- アンカー文字列は LF 表記で持つ。これを守らんとアンカーが一生マッチせん。
- **Phase 1 でビルド工程を入れたら、src/ は LF 固定にして bundle 出力時だけ CRLF にする等、ルールを1箇所に集約する**。今の「全編集でCRLF地獄」を終わらせるのが Phase 1 の隠れた目的の一つ。

### 法則4: ファイルが大きすぎて「普通の編集」ができない
- 17,265行・約740KB。Read で一部しか読めん、Edit のアンカーも巨大ファイルでは扱いづらい。
- 今は **node スクリプトでアンカー一意性を確認(--dry)→一括 splice** という運用でしのいどる(apply_*.js 群がその実例。C:\tmp に残っとるはず)。
- この「編集が物理的に苦痛」という状態こそ Phase 1 でファイル分割する最大の動機。バグ修正より先にこれをやりたくなるが、**Phase 0 のテスト無しに分割するのは禁忌**(法則5)。

### 法則5: 偽リダクションは絶対悪 (黒塗り機能の血の掟)
- 黒塗り(墨消し)で「黒い四角を上に被せただけ・下の文字がコピペで抜ける」は**情報漏洩そのもの**。これを構造的に作らない。
- 正しい実装: 影響ページを200DPIでcanvasラスタ化→fillRectで黒を焼く→JPEG→pdf-lib embed。**pdf-lib の drawRectangle で黒を被せるのは禁止**(下の文字が残る)。
- 出力前に `verifyRedaction` で getTextContent==0 を機械確認、1文字でも残ればDL中止。
- リファクタでこの出力エンジン(generateRedactedPdf/verifyRedaction)を触るときは、**生バイト走査で秘密文字列がゼロ**になることを必ずテストで担保し続ける。

## 1. なぜリファクタが要るのか — 39バグが証明したこと

「無駄が多いから綺麗にしたい」では弱い。今日(2026-06-10)直した39バグの**3分の1は、共通基盤が無いことが直接の原因**やった。これが動かぬ証拠や:

### 証拠A: 座標写像が2実装あるせいで同じバグを2回踏んだ
- 「viewport(回転後)座標 ↔ ページ(回転前)座標」の写像が **imgPlace と黒塗りで別実装**。
- 黒塗りの回転バグ(uniq-3 周辺)を直しても imgPlace の回転バグ(C2)には1ミリも効かず、**別々に設計・別々に検証**した。
- CropBox 対応(H9)も同じ。imgPlace だけ直した。黒塗りには CropBox 対応が無い(まだバグが眠っとる可能性が高い)。
- → **Phase 2 の `viewportToPage()` 単一実装が最重要**。ここを1箇所にすれば、回転・CropBox の事故が全モードで一掃される。テストも一番厚く張る。

### 証拠B: 「エラー即消し」が3モードで同時多発
- `setStatus(err) → 直後に setStatus('')` で読込失敗が一瞬で消えて無言スキップ、が PDF編集・変換・画像→PDF で**コピペ的に3回**起きとった(H7/M4/M5)。
- v3.7.5 で直したが、直し方も `failedNames` 配列を**3モードに別々にコピペ**した。つまり「直した直後から既にコピペ負債」。
- → Phase 2 の StatusBar 部品 + failedNames パターンの部品化で、この種が二度と発芽せん。

### 証拠C: 実行中ガードが4つの別名で乱立
- `isConverting`(変換)/ `st.exporting`(黒塗り出力)/ `st.rendering`(黒塗り描画)/ `isRendering`(imgPlace描画)。
- 全部「処理中はUIを触らせない」という同じ概念。なのにバラバラやから、黒塗りで実行中ガードのバグ(uniq-1)を踏んだとき、その教訓が他モードに伝播せんかった。
- → BusyLock 部品に統合。

### 結論
このコードベースのバグは「個々のミス」やのうて**「同じ部品が無いから同じ間違いをN回する」という構造的なもの**。リファクタ=バグ予防策や。綺麗ごとやない。

## 2. Phase ごとの「なぜこの判断か」と「踏むであろう罠」

### Phase 0 がなぜ全ての前提か
- **テスト無しでこの規模を分割するのは自殺行為**。今日でさえ、1つの修正が他モードを壊してないか手動で何度も確認した。分割は影響範囲が桁違いに広い。
- スモークテストの「種」は既にある: v3.7.4〜3.7.6 と黒塗りの検証で書いた preview_eval のコード群(合成PDFをPDFLibで作る→UI操作を合成イベントで→出力をpdf.jsで読んでピクセル/バイト判定)。これを Playwright のテストファイルに移植するだけ。**ゼロから書くんやない、移植や。**
- **v3.8.0(黒塗り)着地が Phase 0 に入っとる理由**: ファイル分割すると feat/redaction ブランチが現在のmonolith構造に対する差分やから、分割後は永久にマージ不能になる。だから分割前に必ずmainへ入れる。これだけは順序を絶対に守る。

### Phase 0 の罠
- **プレビュー(preview_eval)はこの重いページで頻繁に劣化する**。症状: 「サムネ生成中... 1/3」で固まる、evalが30秒タイムアウト、blobが90バイトの別物を掴む、InvalidPDFException。
  - 対処: `location.reload()` では足りん。**preview_stop → preview_start でブラウザごと再起動**してクリーンな状態で1発で通す。今日これで何度も救われた。
  - だから Playwright 化するときも「各テストは独立したページコンテキストで」が鉄則。状態を持ち越すと劣化を再現してしまう。
- 合成PDFの作り方の定石(テストの心臓):
  - ページ内の `window.PDFLib` を流用して PDF を作る(別途ライブラリ不要)
  - 重複文字の再現は `drawText` を同座標に2回(fill+stroke二度描き相当)
  - 回転は `page.setRotation(degrees(90))`
  - CropBox は `page.setCropBox(x,y,w,h)`
  - EXIF回転JPEGは手で APP1/TIFF/IFD バイトを注入(今日 H4 検証で実際に作った。そのコードがテストの金型になる)
  - 黒塗り検証は「出力を生バイト走査して秘密文字列がゼロ」+「getTextContentが0」
- 出力の捕捉: `URL.createObjectURL` をフックして blob を**即座に永続Uint8Arrayへコピー**(`b.arrayBuffer().then(buf => window.__bytes = new Uint8Array(buf))`)。blobは2回読むとdetachする。複数blobが飛ぶモード(画像5枚など)は**マジックバイトで選別**(`%PDF`/`PK`/`PNG`)。
- `confirm`/`alert` はテスト前に `()=>true`/`()=>{}` で潰す(リロードで戻るので毎回やる)。

### Phase 1 の罠
- build.js を書いたら、**まず「現行bundleと機能等価」を証明してから**src/への分割を始める。順番が逆だと、分割のバグかビルドのバグか切り分け不能になる。
- 取説(guideSource, 3,377行)は `<script type="text/html">` の中身。**この中に絶対 `</script>` を書いてはいけない**(HTMLパーサが type 無視で早期終了する。過去 v3.3.5 で実際に取説全体を破壊した)。分割後のビルドで「連結後に `</script>` が紛れ込んでないか」を機械チェックする gate を build.js に入れる。
- CSS/HTML/JS の連結順序が現行と1バイトでも変わると挙動が変わりうる(CSSカスケード、スクリプト実行順)。**現行bundleの並び順を正準とする**。

### Phase 2 の罠 (一番慎重に)
- 共通化は「1モード移行ごとにスモーク→リリース」。**6モード一気に共通化に乗せ替えるな**。1つずつ。
- `viewportToPage()` 統一は、imgPlace と黒塗りで**座標系の規約が微妙に違う可能性**がある(原点、Y軸、mm/pt、padYの有無)。安易に「同じやろ」で統合せず、両者の現行出力を固定(ゴールデン)して、統合後も**ピクセル一致**することを確認してから乗せ替える。ここが回帰の最大リスク。
- `catch(_e){}` 35箇所の棚卸し: 全部が事故やない。「暗号化PDFは load で throw させて原本返す」のように**意図的な握り**もある(PdfSanitize がそれ)。3分類する: ①notifyWarnすべき(無言失敗)②failedNamesに積むべき③理由コメント付きで握ってOK。**一律で notifyWarn にすると五月蝿くて使い物にならん**。文脈を読め。

### Phase 3 の罠
- imgPlace 4,700行の分割が本丸。**状態(placements, imageLibrary, currentPageIndex...)が全関数から触られとる**から、ファイルを割ると状態の共有方法(モジュールスコープの単一stオブジェクト等)を先に決める必要がある。
- デッドコード削除は**必ず grep で使用箇所ゼロを確認→一覧提示→承認**。血の掟2(削除は承認制)。「使ってなさそう」で消すと、動的に参照されとるやつを消して事故る。

### Phase 4 の罠
- CSS の死にスタイル削除: v3.7.4 で「Phase1残骸の max-width が横長CADを潰す」実害を踏んだ。**未使用に見えても実害を出しとるスタイルがある**。セレクタ単位で「DOMに該当要素が存在するか」を実スキャンしてから消す。

## 3. 検証の作法 (これが無いとリファクタは博打になる)

今日確立した検証パターン。Phase 0 でこれをスクリプト化する。

1. **node --check で構文ゲート**: インライン `<script>`(type無し/jsonでない)を抽出して `node --check`。パースエラーを即検出。編集のたびに最初にやる。
2. **回帰ガード grep**: 編集後にシンボル出現数を数える(`function rectsOverlap` が1か、`movePage` が0か等)。意図せぬ重複定義・残骸を機械検出。apply_*.js の末尾にこのガードを必ず付けた。
3. **アンカー一意性 --dry**: 一括適用前に全アンカーが「ちょうど1回」出現するか確認。0や2なら中止。
4. **プレビュー実機**: クリーンサーバ起動→合成PDF→UI操作(合成イベント、`{bubbles:true}` 必須)→出力をピクセル/バイト判定。
5. **黒塗りだけは追加で**: 生バイト走査 + getTextContent=0 の二重確認。本物消去の証明。

「動いた気がする」では止めん。**ピクセルかバイトで機械判定できるまで**やる。今日それで偽の「FAIL」(透かし社外秘を秘密残骸と誤認、blob detach で空判定)を3回踏んで、毎回掘り下げて真相に辿り着いた。表面の結果を信じるな。

## 4. やってはいけないこと (過去の失敗の墓標)

- `<script type="text/html">` の中に `</script>` を書く → 取説全体が壊れる(v3.3.5)
- 黒塗りで pdf-lib drawRectangle で黒を被せる → 偽リダクション
- main に半端な状態を push → auto-update で全同僚に半端版が配られる(配布ゲートを必ず守る。未完成は feat ブランチ)
- バグ修正とリファクタを同じコミットに混ぜる → 何が原因で壊れたか追えなくなる
- プレビューが劣化した状態で「直ってない」と結論する → サーバ再起動して再検証(環境問題をコードバグと誤診しない)
- ZIP を Compress-Archive で作る → 日本語ファイル名が cp932 化けする。必ず `[System.IO.Compression.ZipFile]::CreateFromDirectory(..., UTF8)`
- リリースで version.json だけ上げて bundle を上げ忘れる(or 逆) → 4ファイル(bundle/version.json/README/ZIP)は常にセット。Phase 0 の release.ps1 で機械化する理由。

## 5. 次の具体的な第一手 (このノートを読んだ人がまず打つ手)

```
状態: main = v3.7.6 (896f068)。配布版は健全。バグ39件修正済み。
      feat/redaction = 黒塗り11修正済み・未マージ・未配布。
```

1. **さっちゃん(ユーザー)に黒塗りの実データテストを促す**。これが Phase 0 の入口。
   - feat/redaction をチェックアウトしてもらい、実物の契約書・図面PDFで黒塗り→出力→「検知ダメ」が改善したか確認。
   - 「小文字が塗れない(MIN_PCT→MIN_PX)」を直したので、ここが一番の改善ポイント。
2. OKなら **feat/redaction を v3.7.6 main にリベース**(main が3.7.1〜3.7.6 進んだ分)。
   - 触っとる領域が違う(黒塗りは redactModule に閉じとる)ので衝突は軽いはず。だが CSS と設定wiringは共有しとるので注意。
   - リベース後にプレビューで黒塗り全シナリオ + 既存5モードの回帰を確認。
3. **v3.8.0 として配布**(APP_VERSION/version.json/README/ZIP、黒塗りタブ初公開)。取説に黒塗りのPARTを追加。
4. ここで初めて **Phase 0 の残り(テスト基盤・release.ps1・死蔵ファイル削除)** に着手。
5. 死蔵ファイル候補(削除前にユーザー承認): `pdf_compact_bundle.html.bak.20260515`(330KB)、`pdf_compressor_guide.html`(126KB旧版)、`HANDOFF_SPEC.md`、`PDF Compact.vbs`(.bat と役割重複か要確認)。

## 6. このプロジェクトの「気持ち」(忘れんといて)

- 使う人は**コードを読めん現場のおっちゃん**。判断材料は「動き・出力・UI」だけ。エラーは日本語で、行動に繋がる言葉で出す(「encrypted」やのうて「保護付きPDFは写真モードで」)。
- 目指す水準は **iPhone標準アプリの気持ちよさ**。「機能が動く」やのうて「触ってて気持ちええ」。リファクタでこの手触りを1ミリも劣化させない。
- 作者(さっちゃん)は一級建築士で本業は現場監督。ツール開発は社内の底上げが目的。**完璧主義やが時間は有限**。リファクタは「いつ止めても損しない」フェーズ設計にしたのは、現場が忙しくなったら中断せざるを得ないことを知っとるから。
