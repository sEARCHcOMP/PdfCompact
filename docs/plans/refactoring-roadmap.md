# PDF Compact 完全リファクタリング計画 (2026-06-10 策定)

17,265行の単一HTMLに蓄積した構造的負債を、**利用者には1ピクセルの変化も見せずに**段階的に解消するロードマップ。
全フェーズ共通の鉄則: **「作り方」を変える。「動き方」は変えない。**

---

## 0. 守るもの (不変条件 — リファクタで絶対に変えない)

| 不変条件 | 理由 |
|---|---|
| 成果物は単一 `pdf_compact_bundle.html` | ダブルクリックで動く・auto-update が1ファイル取得・100%ローカルという製品の根幹 |
| ランタイム依存ゼロ追加 | CDN(pdf.js)+同梱(pdf-lib/fontkit/jsPDF/JSZip)の現構成を維持 |
| フレームワーク/TypeScript/bundler 不採用 | ビルドは「連結のみ」。ツールチェーンの寿命リスクを背負わない |
| 全モードの挙動・見た目 | リファクタ・リリースは機能ゼロ変更。同僚は何も気づかないのが成功 |
| 血の掟 | ファイル削除は一覧提示→承認制。main に半端な状態を置かない |

## 2. フェーズ計画

> 規模感の単位は「セッション」= 今日のような作業1回分。
> **各フェーズの出口 = 機能ゼロ変更のリリース**として配布する(問題があれば1つ前の bundle に戻すだけでロールバック完了)。

### Phase 0: 安全網 (これ無しで分割を始めるのは禁忌) — 1〜2セッション

1. **v3.8.0 を先に着地させる**
   feat/redaction (黒塗り11修正済み) を新main(v3.7.6)へリベース → さっちゃんの実データテスト → マージ → 配布。
   **ファイル分割後はこのブランチが永久にマージ不能になるため、Phase 1 より前が絶対条件。**
2. **自動スモークテスト基盤**
   v3.7.4〜3.7.6 の検証で手書きしたプレビューテスト(合成PDF生成→UI操作→出力をピクセル/バイトで機械判定)をスクリプト化。
   - 形: Node + Playwright (ヘッドレスChromium) で「6モード × 代表シナリオ」のゴールデンテスト
   - 種: **今日直した39バグの検証コードがそのままテストケースの種になる**
   - 黒塗りの「本物消去検証」(生バイト走査で秘密ゼロ) も必須ケース
3. **リリーススクリプト** `build/release.ps1`: 一発で version bump + README履歴 + ZIP再構築(UTF8) + git tag
4. **死蔵ファイル削除** (上記4ファイルの一覧を提示→承認後に削除)

**完了条件**: `npm test` 一発で全モード green / リリースがワンコマンド / v3.8.0 配布済み

### Phase 1: ビルド工程の導入 (構造の要) — 1〜2セッション

```
src/
  html/        … タブ・各モードパネル (分割)
  css/         … base.css + モード別 + components.css
  guide/       … 取説をPART単位で分割
  js/
    lib-loader.js      … CDN/同梱ライブラリ読込
    sanitize.js        … PdfSanitize + 設定 + 更新チェック
    mode-compress.js   … 軽量化
    mode-img2pdf.js    … 画像→PDF
    mode-convert.js    … 変換
    mode-pdfedit.js    … PDF編集
    mode-imgplace.js   … 画像配置 (Phase 3 でさらに分割)
    mode-redact.js     … 黒塗り
build.js               … 依存ゼロの連結スクリプト (node build.js → pdf_compact_bundle.html)
```

- build.js は**連結するだけ**。minify も transpile もしない(出力の可読性=デバッグ性を維持)
- 検証: ビルド出力が現行 bundle と**機能等価**であることをスモーク全通過+目視diffで確認
- 効果: node スプライス編集の廃止 / git diff が読める / モジュール単位の並列作業が可能に / CRLF地獄の終了

**完了条件**: ビルド出力で全テスト green / 以後の全編集が src/ 経由

### Phase 2: 共通基盤の抽出 (コピペ族の殲滅) — 2〜3セッション

新設する共通モジュール(各モードから1つずつ移行、**1モード移行ごとにスモーク→リリース**):

**`js/ui-common.js`**
- StatusBar: setStatus 4定義→1 + failedNames「消えないまとめ表示」パターンを部品化
- BusyLock: 実行中ガード4種→1つの作法 (lock()/unlock()/guard(fn)) — 黒塗りの st.exporting/st.rendering、変換の isConverting を統合
- DropZone: 25箇所の dragover 配線→1部品
- download(blob, name) / uniqueFilename / ZIP出力 / 成功モーダル

**`js/pdf-common.js`**
- detectProtectedPdf(bytes): 暗号化検出の正準実装 (pdf.js getPermissions 方式) — 4モードのバラバラ対応を統一
- viewportToPage(page, rect): **座標写像の単一実装** (回転0/90/180/270 + CropBox原点。imgPlace と黒塗りの2実装を置換。これが一番事故が多かった領域なので最重点テスト)
- readExifOrientation(file): EXIF 2実装→1
- canvas系: flattenWhite / 寸法キャップ(8192px) / loadImageElement (6複製→1)

**完了条件**: 上記の重複実装ゼロ(grep で定義各1) / catch(_e){} 35箇所を棚卸しして「notifyWarn する / failedNames に積む / 理由コメント付きで握る」の3択に分類

## 3. 進め方のルール

1. **順序固定は 0→1 のみ** (テスト無しの分割は禁忌)。Phase 2以降は状況で入れ替え可
2. 各フェーズは独立して価値が完結する — **どこで止めても損しない**
3. フェーズ中の新機能開発は凍結。やるならフェーズ境界で
4. リファクタ中にバグを見つけたら**直さず記録** → 別コミット・別リリースで修正 (挙動変更とリファクタを同じコミットに混ぜない)
5. 各リリース前: スモーク全通過 + 手動で1モード抜き打ち確認
6. コミット粒度: 「1コミット = 1モジュール移行」を上限に

## 4. リスクと対策

| リスク | 対策 |
|---|---|
| 分割ミスで動くものを壊す | Phase 0 のテストが防波堤。各リリースは bundle 1ファイル差し替えで即ロールバック可 |
| feat/redaction が宙に浮く | Phase 0 で最優先着地 (これだけは順序を絶対に守る) |
| ビルド忘れで古い bundle を配布 | release.ps1 がビルド→テスト→ZIP→タグまで一気通貫、手動配布を廃止 |
| リファクタ疲れで中途半端に止まる | フェーズ単位で完結する設計 + 各フェーズ出口で「動く main」を維持 |
| AIセッション間の文脈喪失 | 各フェーズの進捗を docs/plans/refactoring-progress.md に記録 (handoff 運用と同じ) |
